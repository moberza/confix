Confix: A Build Tool on Top of GNU Automake

Introduction

Confix does what is otherwise the job of the package maintainer: write input files for Automake and Autoconf. By examining the package source, it determines how to shape configure.in and the different Makefile.am files in the package in order to perform a clean build.

The main features of Confix include

  • Dependency calculation. A correct build order of the package's modules is automatically determined. Furthermore, installed packages integrate into dependency calculation by means of a global "repository".
  • Propagation of Autoconf checks between packages. When one package uses another package, there often is the problem that the using package needs to apply the same Autoconf checks (or a subset thereof) that were applied when the other package was built. Not doing so can lead to subtle bugs; for example, disagreement about member offsets in structures. Confix tries to solve that problem by propagating checks into the using package.
  • Mapping of C++ namespaces to include paths. As Confix examines the source code, it can detect C++ namespaces, and map that hierarchy to the install location of the respective header files. (This feature can be suppressed though.)
  • Interfaces for custom build instructions. The plan is that a package maintainer need not write any build instructions at all, but to let Confix do that task by scanning the source code. However, should Confix need help from the maintainer, this can be done using a Python interface to modify build properties. One can place arbitrary Python code into source files, for example, or into dedicated configuration files.

Releases and Download

You can download releases from the project summary page at Sourceforge. The most recent stable release is 1.4. If no release is recent enough, you may always checkout a development version from CVS (follow the instructions found on that page, substituting confix for modulename). You can also browse the CVS repository without using CVS.

We have tried to define a numbering scheme for releases which may help you decide if you should install a particular version. See below for the numbers and more.

Installation

So far, Confix is only available as a source package (which is better than nothing though). Confix uses Python's Distutils for distribution and setup. If you're not already familiar, here's how to do it.

  • Unpack the file.
    $ pwd
/home/jfasch
$ tar zxplf Confix-x.y.z.tar.gz
  • Install the package using Distutils.
    $ cd Confix-x.y.z
$ pwd
/home/jfasch/Confix-x.y.z
$ python setup.py install --prefix=${HOME}/sandbox
  • Modify your environment accordingly. For example, given the above prefix, add the following to your .bashrc.
    PREFIX=${HOME}/sandbox
export PYTHONPATH=${PYTHONPATH}:${PREFIX}/lib/python2.2/site-packages
export PATH=${PATH}:${PREFIX}/bin

You can find more detailed installation instructions in the documentation.

Release Numbering, CVS

We try not to roll dice when we release a new version, but adhere to the following numbering scheme.

  • Release numbers are <major>.<minor>.<bugfix>.
  • The <bugfix> version is incremented when there have been bug fixes from previous releases with the same <major>.<minor> version. Of the three parts of the version, the bugfix part is supposed to change most often. It is considered safe to upgrade to a release that is only higher in its bugfix version as there are no differences in any generated files (except for differences that are the target of one particular bug fix, of course).
  • The <minor> version is incremented with a release which contains significant new features, or when there are differences in generated files that make them incompatible with previous versions of Confix. The emphasis here is on the word generated, as a minor version increment does not require you to modify any source files. The most user-visible incompatibility will probably be that of the module description files. (See the documentation for what this is.)
  • The <major> version is incremented when users are required to modify source code. You are encouraged to check the release notes of that release before you decide to switch.

The connection between releases and CVS is as follows.

  • Every release is tagged. The name of the release tag is RELEASE-<major>_<minor>_<bugfix>.
  • New features and incompatible changes are developed on the main trunk, and this is where the release is made from. (What follows is that all zero-bugfix releases are made from the trunk.)
  • Bugfixes to a zero-bugfix release make for a bugfix branch. That branch is rooted at the release tag of that release, and is named RELEASE-<major>_<minor>_0-bugfix. All bugfix releases to the <major>.<minor> release are done on that branch.
